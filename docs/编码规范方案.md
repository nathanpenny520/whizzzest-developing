# 焰境·万载 编码规范方案

> 基于 2026-05-26 全面代码审计，覆盖 contracts / backend / frontend 三个层面。
> 每条规则附带当前问题位置和推荐修复方式。优先从 P0 开始治理。

---

## P0 — 立即修复（影响正确性和安全性）

### 1. `api/aiChat.ts` 绕过 auth 拦截器

**问题**：`packages/frontend/src/api/aiChat.ts:21` 使用原始 `axios.post()` 而非从 `client.ts` 导出的 `api` 实例。AI 请求没有 Bearer token，也不会触发 401 自动刷新。

**规则**：所有 API 调用必须经过 `@/api/client` 导出的 `api` 实例。

```ts
// ❌ 当前
import axios from 'axios'
const response = await axios.post(aiChatUrl, body, { timeout: 180000 })

// ✅ 应为
import { api } from '@/api/client'
const response = await api.post('/ai/chat', body, { timeout: 180000 })
```

### 2. `ai.service.ts` 使用裸 `new Error()`

**问题**：`packages/backend/src/modules/ai/ai.service.ts:67` —— 唯一一处用原生 `Error` 抛异常，产生未格式化的 500 响应。

**规则**：后端所有异常必须使用 NestJS 内置异常类或自定义 `HttpException`。

```ts
// ❌ 当前
throw new Error('Upstream API error: ...')

// ✅ 应为
throw new HttpException(
  { code: ErrorCode.INTERNAL_ERROR, message: '...', data: null },
  HttpStatus.INTERNAL_SERVER_ERROR,
)
```

### 3. `jwt-auth.guard.ts` 放错了位置

**问题**：`packages/backend/src/modules/auth/jwt-auth.guard.ts` —— 被 8 个模块引用的守卫放在 auth 模块内。

**规则**：被多个模块共用的基础设施文件放在 `common/` 目录下。auth 专用的（如 `jwt.strategy.ts`）留在 auth 模块内，auth 无关的（如 `jwt-auth.guard.ts`）移到 `common/guards/`。

```
✅ 移动后：
common/guards/
  jwt-auth.guard.ts     ← 从 modules/auth/ 移入
  optional-auth.guard.ts
  roles.guard.ts
```

### 4. `ROLES_KEY` 重复定义

**问题**：`packages/backend/src/common/decorators/roles.decorator.ts:4` 和 `packages/backend/src/common/guards/roles.guard.ts:5` 各定义了 `ROLES_KEY = 'roles'`。

**规则**：常量只定义一次，其他地方 import。

## P1 — 类型规范（contracts 的正确使用）

### 5. contracts 是唯一真相源

**问题现状**：contracts 定义 27 个类型，20 个（74%）从未被使用。前端 0 引用，后端只用了 AI 和 firework 两个模块的类型。

**规则**：

- 一个类型如果前后端都用到 → 放 contracts
- 一个类型如果只有后端用到 → 放后端模块的 `dto/` 或 `interfaces/`
- 一个类型如果只有前端用到 → 放前端的 `types/`
- contracts 不需要"预测未来"——暂时没人用的类型就删掉

### 6. 前端必须从 contracts 引用共享类型

**规则**：以下已存在于 contracts 的类型，前端不得本地重定义：

| contracts 类型           | 前端重复处                                     |
| ------------------------ | ---------------------------------------------- |
| `ChatRequest`            | `types/aiChat.ts:38`                           |
| `ChatMessage`            | `types/aiChat.ts:4`                            |
| `AIActionType`           | `types/aiChat.ts:31`                           |
| `IAIResponse`            | `types/aiChat.ts:20` (结构不同，需统一后导入)  |
| `ApiResponse<T>`         | `api/aiChat.ts:8` (`NestResponse`)             |
| `IFireworkRecipeSummary` | `FireworkLeaderboard.vue:72` (`RecipeSummary`) |
| `UserRole`               | 后端 7 处从 `@prisma/client` 导入              |

```ts
// ✅ 统一后
import type { ChatRequest, ChatMessage, IAIResponse } from '@wanzai/contracts'
```

### 7. 命名规范：接口一律 I 前缀

**问题**：contracts 的 `ai.ts` 中 `IAIResponse` / `IAIAction` 带 I 前缀，但 `ChatMessage` / `ChatRequest` / `SSEEvent` 不带。

**规则**：所有 interface 使用 `I` 前缀。enum 和 type alias 不用。前后端统一。

```ts
// contracts
export interface IChatMessage { ... }    // ✅
export interface IChatRequest { ... }    // ✅
export interface ISSEEvent { ... }       // ✅
export type AIActionType = ...           // ✅ type alias 不前缀
export enum UserRole { ... }             // ✅ enum 不前缀

// 前端
interface ICommentItem { ... }           // ✅
interface ICouponData { ... }            // ✅
```

### 8. `UserRole` 必须来自 contracts

**问题**：后端 7 个文件从 `@prisma/client` 导入 `UserRole`，与 contracts 的 `UserRole` 是两套独立的枚举。

**规则**：Prisma schema 保留 enum（数据库需要），但应用层代码统一从 contracts 导入。在值转换边界做一次映射。

### 9. `ErrorCode` 枚举必须使用

**问题**：contracts 定义了 `ErrorCode` 枚举（`SUCCESS = 0, BAD_REQUEST = 40000, ...`），但零引用。后端遍地魔法数字。

**规则**：所有错误码和成功码统一使用 `ErrorCode` 枚举。`docs.controller.ts:42` 的 `404`（3 位）与其它文件 `40400`（5 位）不一致也必须统一。

## P2 — 后端架构规范

### 10. Controller 不写业务逻辑

**问题分布**：

| 文件                             | 问题                               |
| -------------------------------- | ---------------------------------- |
| `auth.controller.ts:24-73`       | 输入验证、try-catch 重建、DTO 映射 |
| `docs.controller.ts:62,100-131`  | 业务级校验、multer 配置内联        |
| `analytics.controller.ts:17-19`  | 请求头解析                         |
| `user.controller.ts:19-21,39-41` | 敏感字段剥离                       |
| `coupon.controller.ts:31-44`     | 跨 service 编排                    |
| `firework.controller.ts:47-49`   | query 参数类型转换                 |

**规则**：Controller 只做三件事 —— 路由装饰器、参数提取（`@Body` / `@Param` / `@Query`）、调用 service 并返回。其他全进 service。

```ts
// ❌ Controller 中做验证
if (!email || !email.includes('@')) throw new BadRequestException(...)

// ✅ 用 class-validator DTO + ValidationPipe
export class SendCodeDto {
  @IsEmail()
  email: string
}

// ❌ Controller 中做跨 service 编排
const merchant = await this.merchantService.findByUserId(user.id)
return this.couponService.create(merchant.id, body)

// ✅ Service 封装编排
// coupon.controller 只调 couponService.createForMerchant(user.id, body)
```

### 11. 全局异常过滤器

**问题**：没有 ExceptionFilter，NestJS 默认的 `{ statusCode, message }` 格式与前端期望的 `{ code, data, message }` 不一致。`auth.controller.ts` 手写 try-catch 做转换是个 workaround。

**规则**：创建 `common/filters/http-exception.filter.ts`，在 `app.module.ts` 注册为 `APP_FILTER`。统一输出格式：

```ts
// 全局异常过滤器输出
{ code: number, data: null, message: string }
```

这样每个 controller/service 只需 `throw new NotFoundException('...')`，无需手动包装。

**附带清理**：删除 `auth.controller.ts:30-43` 的手动 try-catch 转换逻辑。

### 12. 模块文件结构统一

**问题**：没有模块有 `dto/` 子目录，DTO 都内联在 controller 里。`knowledge/prompts.ts` 被跨模块引用（ai 模块引用 knowledge 模块的文件）。

**规则**：每个模块标准结构：

```
modules/xxx/
  xxx.module.ts
  xxx.controller.ts
  xxx.service.ts
  dto/                  ← 按需（有 DTO 时创建）
    create-xxx.dto.ts
    update-xxx.dto.ts
```

跨模块共享的常量/工具移入 `common/` 或独立为 `shared/` 子目录。`knowledge/prompts.ts` 被 ai 模块引用 —— 要么移入 common，要么通过 `KnowledgeModule` 的 exports 暴露。

### 13. Service 避免直接跨表操作

**问题**：`merchant.service.ts:26-31` 直接 `this.prisma.user.update()` —— MerchantService 操作 User 表，绕过了 `UserService` 的封装。

**规则**：优先通过目标模块的 service 操作数据。如果 `UserService` 没有 `promoteRole()` 方法就加一个。

```ts
// ❌ MerchantService 直接操作 User 表
await this.prisma.user.update({ where: { id: userId }, data: { role: 'MERCHANT' } })

// ✅ 通过 UserService
await this.userService.promoteToMerchant(userId)
```

## P3 — 前端架构规范

### 14. 创建 API 层（消除组件直调 raw api）

**问题**：10+ 个 `.vue` 文件直接 `import { api } from '@/api/client'` 然后裸调 `api.get('/merchants')` 等。目前 `api/` 目录只有 `client.ts` 和 `aiChat.ts`。

**规则**：每个资源创建对应的 API 模块：

```
api/
  client.ts           ← axios 实例
  aiChat.ts           ← AI 聊天（修复后使用 api 实例）
  fireworks.ts        ← GET/POST/PUT/DELETE /fireworks
  coupons.ts          ← GET/POST/DELETE /coupons, claim, redeem
  merchants.ts        ← GET/POST/PUT /merchants, verify
  comments.ts         ← GET/POST/DELETE /comments, like
  docs.ts             ← GET/POST/PUT/DELETE /docs, upload
  users.ts            ← GET /users
  analytics.ts        ← POST /analytics, GET stats
  knowledge.ts        ← GET/POST/PUT/DELETE /knowledge
```

每个模块封装 `ApiResponse<T>` 的解包。组件只调用封装好的函数。

```ts
// api/fireworks.ts
import { api } from './client'
import type { ApiResponse, IFireworkRecipe } from '@wanzai/contracts'

export async function getFireworkRecipes(sort?: string) {
  const res = await api.get<ApiResponse<IFireworkRecipe[]>>('/fireworks', { params: { sort } })
  return res.data // 已经在 client interceptor 中解过一层，这里拿到 { code, data, message }
}
```

### 15. 组件行数上限

**问题**：15 个 `.vue` 文件超过 300 行，FireworkPage.vue 达到 2967 行。

**规则**：

| 类别       | 上限   | 超限处理方式           |
| ---------- | ------ | ---------------------- |
| 页面组件   | 500 行 | 抽 composable + 子组件 |
| 功能组件   | 300 行 | 抽 composable + 子组件 |
| 展示组件   | 150 行 | 拆子组件               |
| composable | 200 行 | 拆多个 composable      |

- 超过 200 行的 composable 拆分
- 少于 20 行且只有一个调用点的 composable 内联回去

### 16. 消除重复代码（Top 5）

| #   | 重复模式                                   | 出现次数     | 修复方式                                                     |
| --- | ------------------------------------------ | ------------ | ------------------------------------------------------------ |
| 1   | `isZh` computed                            | 7 文件       | 统一放入 `useLocalizedPath` composable 或导出独立 composable |
| 2   | axios 错误提取 `e.response?.data?.message` | 7 文件       | `@/utils/error.ts` 导出 `extractErrorMessage(e, fallback)`   |
| 3   | `localizedPath()` 函数                     | 3 文件       | 统一使用已有的 `useLocalizedPath` composable                 |
| 4   | `loading/error/actionLoading` 三元组       | 10+ 文件     | 可选：创建 `useAsyncData` composable                         |
| 5   | Coupon 类型定义                            | 3 处不同定义 | 统一到 `types/coupon.ts`，基于 contracts 的 `ICoupon` 扩展   |

### 17. 类型集中管理

**问题**：`types/` 目录只有 `aiChat.ts` 和 `amap.d.ts`。大量接口散落在 `.vue` 文件和 composable 中。

**规则**：前端专用类型放 `types/`，一人一地：

```
types/
  aiChat.ts        ← 已存在，清理重复后只保留 UI 专用类型
  amap.d.ts        ← 已存在
  coupon.ts        ← CouponCard.vue:22 的 CouponData 移入
  comment.ts       ← CommentSection.vue:164-180 的 CommentAuthor/CommentItem 移入
  firework.ts      ← FireworkLeaderboard.vue:72 的 RecipeSummary 移入
  doc.ts           ← DocsPage.vue:95 和 DocDetailPage.vue:77 合并为一个 Doc 类型
  merchant.ts      ← MerchantPage.vue:486 的 MerchantImage 移入
  user.ts          ← auth.ts:29 的 UserProfile 移入
```

### 18. 统一事件通信

**问题**：`auth.ts:155` 用 `window.dispatchEvent(new CustomEvent(...))` 而不是项目已有的 mitt 事件总线。

**规则**：跨组件通信统一用 `@/eventBus` 的 mitt emitter。禁止裸 `window.dispatchEvent`。
